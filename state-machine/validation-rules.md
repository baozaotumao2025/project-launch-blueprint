# Validation Rules

> 用来判断 `state machine map` 能不能进入下一阶段。

## 1. Validation Strategy

状态机验证分两层：

1. 正向测试：确认状态机确实覆盖了关键流程的阶段变化和业务事件。
2. 负向测试：主动寻找漏状态、假事件、假守卫、错误终态、跨流程混写和 UI 化命名。

辅助证据只能用来核验和补漏，不能把它们再次抽象成新的主流程。

LLM 审查时必须同时执行两层验证。只做正向通过，不算通过。

验证必须遵循顺序：
1. 先看来源
2. 再看状态语义
3. 再看事件和守卫
4. 再做反例攻击
5. 最后决定回退或通过

任何一步不能明确说明，都视为未通过。

## 2. Pass Table

| Check | Look At | Pass When | Fail When |
| --- | --- | --- | --- |
| Traceability | `discovery capability map` / `domain model map` / 辅助证据 | 每个状态机元素都能回指业务能力、领域对象或流程例子 | 只能靠推断，回不去上游产物 |
| State Purity | state names | 状态是业务阶段，不是页面、按钮、组件或数据字段 | 把 `TabOpen`、`SaveClick`、`FormDirty` 当状态 |
| Event Purity | events | 事件是已发生的业务事实，不是操作命令 | 把点击、提交、刷新、请求写成事件 |
| Guard Clarity | guards | 每个守卫都能说清前置条件和禁止条件 | 守卫模糊、只写“应该”“可能” |
| Transition Validity | transitions | 每条转移有明确起点、终点、触发和约束 | 任意跳转、缺少前提、终点不清 |
| Terminal Validity | terminal states | 终态能解释为什么结束以及结束后行为 | 把任意收尾都写成终态 |
| Flow Cohesion | flow scope | 一个状态机只覆盖一条连贯业务流程 | 同时混入注册、报价、收资料、签字 |
| Testability | examples | 每个状态和转移都能写出可验证例子 | 写不出 Given / When / Then |
| Boundary Fit | domain model | 状态变化与领域边界一致，不越界 | 一个状态机硬塞多个领域 context |
| Evolution Fit | model shape | 状态可拆分、可合并、可重命名 | 过早锁死实现细节 |
| Positive Test Evidence | examples | 每个状态机元素都能给出可执行的正向验证样例 | 只有结论，没有可判定样例 |

## 3. Positive Test Matrix

| Signal | Good Example | Why It Passes |
| --- | --- | --- |
| 清晰阶段 | `待审核 -> 待建档 -> 待报价` | 是业务阶段，不是 UI 变化 |
| 明确事件 | `客户确认报价` | 是已发生的业务事实 |
| 明确守卫 | `资料齐全且来源已核验` 才能进入下一步 | 能判断是否允许转移 |
| 明确终态 | `已完成` 表示业务闭环 | 终态能解释业务结束 |
| 单流程 | 只建模“新客接纳流程” | 一条连贯主线，边界清楚 |
| 能写例子 | Given 资料齐全 When 会计师确认 Then 进入下一状态 | 可测试、可回放 |

## 4. Negative Test Matrix

| Failure Mode | Bad Example | Why It Fails |
| --- | --- | --- |
| 页面化状态 | `InvitePageOpen` | 这是页面展示，不是业务状态 |
| 按钮化事件 | `SubmitButtonClicked` | 这是交互动作，不是业务事实 |
| 数据字段化状态 | `DocumentStatus = red` | 这是视觉或字段值，不是业务阶段 |
| 假守卫 | `if(true)` | 没有约束，不是守卫 |
| 漏状态 | 从 `待审核` 直接跳到 `已完成` | 缺少关键阶段 |
| 错误终态 | `已上传资料` 被当终态 | 业务并未结束 |
| 跨流程混写 | 一个状态机同时包住接客、建档、归档 | 流程边界被打散 |
| 逆向转移失控 | `已完成 -> 待建档` 没有重开规则 | 破坏流程稳定性 |
| 纯推断 | 没有辅助证据或上游映射佐证 | 没有证据链 |

## 5. LLM Review Questions

审查状态机时，LLM 必须逐项回答：

1. 这个状态对应哪条已验证 capability 或哪段流程？
2. 这个状态名是否只是页面或组件名？
3. 这个事件是否真的是业务事实，而不是点击或请求？
4. 这个守卫是否能写成明确前置条件？
5. 这个终态是否真的代表业务结束？
6. 这个状态机是否只覆盖一条连贯流程？
7. 是否存在漏状态或虚假跳转？
8. 是否存在跨领域、跨阶段混写？
9. 是否能构造一个最小反例击穿它？
10. 如果去掉页面名和按钮名，它是否还成立？

如果任一问题无法被明确回答，就视为不通过。

## 6. Go Rules

- 至少有一条来自 `discovery capability map` 的能力佐证。
- 至少有一条来自 `domain model map` 的领域对象或边界佐证。
- 辅助证据只用于核验，不得变成新的主输入。
- 每个状态都有业务语义，不是 UI 语义。
- 每个事件都有业务事实，不是操作语义。
- 每个守卫都能解释允许与禁止。
- 每个终态都能解释结束条件。
- 每个流程都能写出可验证例子。
- 每个状态机都至少有一个正向测试和一个负向测试

## 7. No-Go Rules

出现以下任一情况，直接回退：

- 证据不足
- 状态语义不纯
- 事件语义不纯
- 守卫不明确
- 终态不成立
- 流程混写
- 边界越界
- 不能构造可执行反例
- 不能明确说明回退点
- 辅助证据被当成了新的主分析对象
