# Step 01: Skill Installation And Artifact Contract

> 目标：把“用户安装 skill 之后，项目目录里应该出现什么、不会污染什么、最终交付什么、哪些临时物也要跟着进 GitHub”定义成一份可执行契约。

## Background

这个 skill 的职责不是停留在文档层，而是把已验证的中间产物一路推进到真实原型代码。

它的运行方式是：

- 前面阶段负责定义、验证和收敛语义
- `implementation` 负责把这些语义映射成目录、文件、组件、服务、测试和页面
- 最终结果必须落到用户仓库的真实代码树里，并作为可提交产物保留

所以我们先把两个空间分开：

- `skill space`：安装在 Codex 全局里的只读能力包
- `project space`：用户当前仓库里的正式代码、阶段文档和运行态记录

## 1. 需要拷贝到用户项目目录的文件

建议只拷贝“项目级最小发行包”，不要把全局 skill 本体整包复制进去。

### 1.1 顶层入口

这些文件负责让用户一眼看懂这套 skill 的目标和使用方式：

- `README.md`
- `index.md`
- `template-spec.md`
- `cli-architecture.md`
- `workflow-state.md`
- `command-reference.md`

### 1.2 阶段目录

这些目录负责承载每一层的契约、验证和样例：

- `discovery/`
- `domain-model/`
- `state-machine/`
- `api-contract/`
- `design-system/`
- `vertical-slice/`
- `quality-gates/`
- `implementation/`

每个阶段目录至少包含：

- `index.md`
- `method.md`
- `validation-rules.md`
- `output-schema.md`
- `prompt-templates.md`
- `example-output.md`
- `checklist.md`

### 1.3 运行态与投影目录

这些文件主要服务于规划、审查和状态追踪，不是最终原型代码本体：

- `.project-launch-blueprint/projections/`
- `.project-launch-blueprint/state/`
- `.project-launch-blueprint/review/`
- `.project-launch-blueprint/manifest/`

如果某些目录只用于本地运行，也要在文档里明确它们是“可丢弃”还是“必须提交”。

但要特别注意：

- `implementation` 生成的原型代码不是这里的临时物
- 只要是最终代码树的一部分，就应该进入正常源码目录并被 Git 追踪

## 2. 如何保证这些文件不污染用户项目

核心原则是“单一命名空间 + 明确分层 + 可回退”。

### 2.1 单一命名空间

所有 skill 运行态文件统一放进 `.project-launch-blueprint/`，不要散落到用户源码目录根部。

这样可以保证：

- 不和用户已有业务目录冲突
- 不污染框架约定目录
- 删除时容易清理

### 2.2 只读模板和可写实例分离

把 skill 自带内容当作只读模板，把用户仓库中的生成结果当作可写实例。

规则是：

- skill 包里的模板不直接改
- 用户项目里的生成结果可演化
- 每次生成都记录 revision
- 上游变更触发下游 stale 标记
- `implementation` 最终写入的是项目正式代码目录，不是只放在 scratch 或 tmp 里

### 2.3 生成前先做命名校验

每次写入用户项目之前都检查：

- 目录是否已存在
- 是否覆盖用户手写文件
- 是否和现有项目同名冲突
- 是否需要显式 `--force`

### 2.4 只追加，不隐式覆盖

默认策略应该是：

- 新文件可创建
- 已有文件不自动覆盖
- 真要覆盖时必须先产出 diff 或预览

### 2.5 可回退

每一步生成都要保留回退点，例如：

- 生成前快照
- 变更清单
- 失败时的回滚说明

## 3. 用户利用这个 skill 的最后生成物是什么

最后生成物不应该只是“一个文件集合”，而应该是一个可交付的项目状态。

建议把最终生成物定义为：

- 一个已经落盘的项目级 skill scaffold
- 一套可追踪的阶段文档
- 一组可验证的实现任务
- 一份可提交到 GitHub 的变更集

更具体一点，最终交付至少应包括：

- 说明文档
- 阶段规范
- 产物映射
- 验证规则
- 实现桥接
- 可选的运行态投影

如果这个 skill 的目标是“帮用户从想法走到可发布工程”，那么最终生成物应该是：

- 可继续迭代的仓库结构
- 可被 Codex 再次读取的阶段状态
- 可被 Git 追踪的源文件和必要产物

## 4. 如何保证临时生成物也能推送到 GitHub

这里要先区分两种“临时”。

### 4.1 真正的临时物

这类文件只用于执行过程，不应该进仓库，例如：

- 中间缓存
- 临时日志
- 一次性审查草稿
- 运行时 scratch file

这类文件应放在：

- `.project-launch-blueprint/tmp/`
- 系统临时目录

并且加入忽略规则，避免误提交。

### 4.2 必须随仓库走的生成物

这类文件虽然是“生成出来的”，但它们是项目资产，应该被 Git 追踪。

做法是：

- 让生成结果直接落到仓库内的正式目录
- 不把这些目录写进 `.gitignore`
- 在 manifest 里记录它们是“生成产物”而不是手工产物
- 提交时把它们当作正常源文件一起提交

这也包括 `implementation` 产生的原型代码、页面、组件、测试和支撑文件。

### 4.3 推荐策略

最稳妥的办法是把“需要推送的生成物”定义为仓库正式内容，把“真正的临时物”定义为本地缓存。

也就是说：

- `可提交生成物` -> 放在项目正式目录，纳入 Git
- `不可提交临时物` -> 放在 `.project-launch-blueprint/tmp/` 或系统临时目录

这样 GitHub 上最终看到的就是完整可复现的项目状态，而不是漏掉半截中间结果。

换句话说，`implementation` 的目标不是产出“可选缓存”，而是产出“要交付、要提交、要继续迭代”的原型代码。

## 5. 本步验证标准

这一步算通过，必须同时满足：

- 能列出要复制到用户项目的文件和目录
- 能明确哪些文件默认不覆盖用户项目
- 能说清最终生成物是什么
- 能区分“要提交的生成物”和“不能提交的临时物”
- 能说明 GitHub push 时哪些内容应该被保留
